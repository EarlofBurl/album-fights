# Architecture Overview

## Philosophy

Album Fights follows a pragmatic, lightweight MVC-inspired pattern without a heavy framework. The goal is to keep the codebase readable, testable, and safe while staying deployable as a single Docker container or desktop app.

---

## Directory Layout

```
src/
  Core/
    Config.php          # Path resolution (web vs desktop), settings load/save
    Security.php        # CSRF generation/validation, typed input helpers, slugify
  Repository/
    AlbumRepository.php # CSV read/write for elo_state + listening_queue
    SettingsRepository.php # JSON read/write for settings
  Service/
    EloService.php      # Elo calculation, matchmaking algorithm, rank classes
    MetadataService.php # Multi-source metadata fetcher (Subsonic -> Last.fm -> ListenBrainz -> iTunes)
    ImportService.php   # API imports, CSV parsing, candidate encoding
    StatsService.php    # Aggregates stats from cache metadata + album data
    AiService.php       # OpenAI & Gemini prompt wrappers
    DuplicateService.php # Fuzzy duplicate detection & merging
  Utils/
    HttpClient.php      # Reusable HTTP/JSON client with desktop SSL bypass
    SubsonicClient.php  # Subsonic API wrapper (search, album lists, cover art)
    CsvHelper.php       # Atomic CSV read/write with backup rotation
templates/
  partials/
    header.php          # Shared HTML head + nav + CSS
    footer.php          # Closing tags
  duel.php              # Duel view (index.php)
  list.php              # Sortable/paginated list + top-25 grid
  stats.php             # Stats dashboard cards
  import.php            # Import forms + candidate table
  queue.php             # Queue management table
  bootcamp.php          # AI boot camp view
  database.php          # Tag blacklist + duplicate resolver
  settings.php          # Settings form with weight sliders
includes/
  autoload.php          # PSR-4 autoloader for App\ namespace
  config.php            # Session start + bootstrap
```

## Request Lifecycle

1. **Entry Point** (`index.php`, `list.php`, …)
   - `require_once 'includes/config.php'`
   - Instantiates `Config`, repositories, services
   - Reads `$_GET` / `$_POST` via `Security` helpers
   - Calls `Security::requirePost()` for mutating actions (CSRF check)
   - Delegates business logic to services
   - Renders template from `templates/`

2. **Services**
   - Stateless (or nearly stateless) classes that implement business rules
   - Services may depend on repositories and other services
   - No direct access to `$_GET` / `$_POST` / `$_SESSION`

3. **Repositories**
   - Only layer that touches the filesystem (CSV/JSON)
   - `AlbumRepository` uses `CsvHelper` for atomic writes with `.bak1/2/3` rotation
   - `SettingsRepository` delegates to `Config`

4. **Templates**
   - Pure presentation; no DB access or business logic
   - Receive pre-computed variables from the entry point
   - Include `partials/header.php` and `partials/footer.php`

---

## Security Model

- **CSRF**: Every `<form method="POST">` includes a hidden token generated by `Security::csrfField()`. Mutating endpoints call `Security::requirePost()`.
- **Input**: All request data is accessed via `Security::getString()`, `getInt()`, `getFloat()`, `getBool()`.
- **Images**: `serve_image.php` validates the file parameter against a strict regex (`/^album_[a-f0-9]{32}\.jpg$/`) and only serves from the cache directory.
- **Passwords / API keys**: Stored plaintext in `data/settings.json`. Ensure the `data/` directory is not directly web-accessible in custom hosting setups (in Docker it is outside the web root by default).

## Data Flow

### Duel Vote
```
index.php (POST)
  -> Security::requirePost()
  -> EloService::calculateResult($a, $b, $score)
  -> AlbumRepository::saveElo($albums)
  -> (Optional) AiService::triggerNerdComment()
  -> template/duel.php
```

### Metadata Fetch
```
MetadataService::getAlbumData($artist, $album)
  -> Check local JSON cache (1-week cooldown)
  -> SubsonicClient::fetchAlbumData()    [if configured]
  -> HttpClient -> Last.fm API           [if key configured]
  -> HttpClient -> ListenBrainz/MB       [if configured]
  -> HttpClient -> iTunes                [fallback]
  -> Write JSON cache
```

### Import
```
import.php (POST)
  -> ImportService::fetchCandidates($source, $mode, $user, $limit)
     -> source-specific API client
     -> filter against existing albums
  -> template/import.php (renders candidate table)
  -> import.php (POST, import action)
     -> ImportService::importToDb() or importToQueue()
```

## Key Design Decisions

1. **No Composer / No framework**
   - A custom PSR-4 autoloader keeps the project dependency-free and easy to ship as a Docker image or desktop app.

2. **CSV as primary database**
   - Human-readable, easy to back up, no database server required.
   - Writes are atomic (`tmp` -> rename with backup rotation) to avoid corruption.

3. **Desktop path resolution in Config**
   - `Config::resolvePaths()` detects Electron/AppImage/Flatpak/Windows via environment variables and switches `data/` and `cache/` paths accordingly.

4. **Services over helpers**
   - Previously duplicated logic (e.g. Subsonic API calls in both `api_manager.php` and `import.php`) is now centralized in `SubsonicClient` and `ImportService`.

5. **Templates extracted from entry points**
   - Entry points are now thin controllers (< 150 LOC each), making them easier to read and safer to modify.

---

## Extension Points

- **New metadata source**: Add a private method in `MetadataService` and call it in `getAlbumData()`.
- **New import source**: Add a fetch method in `ImportService` and wire it into `fetchCandidates()`.
- **New stats**: Extend `StatsService::buildStats()` and the `templates/stats.php` card grid.
- **New page**: Create a thin entry point + template following the existing pattern.
